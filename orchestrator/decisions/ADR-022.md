## M4.11 — Cross-country trend gap detector (✅ shipped)

**Mapea a la cita #2 del audio.**

### Diseño

- Usa el `country_focus` que `idea_analyzer` ya rellena en `analysis_json`
  desde M3.11 — cero costo adicional de LLM.
- Cluster heurístico por `suggested_topic[:60]` (en M5 reemplazable por
  embeddings vectoriales si los volúmenes lo justifican).
- Por cada cluster, detecta países "validados" (≥2 signals + ≥1 👍 o
  ≥2 signals con score ≥ 0.7) y mapea los países objetivo donde NO existe
  ningún signal del mismo cluster.
- `opportunity_score = clamp((n_val × 0.3) + (n_ups × 0.2) + (n_missing × 0.05), 0, 1)`.

### API

- `GET /api/v1/trend-gaps?min_validation_signals=2&min_validation_feedback=1&countries=Ecuador,Colombia,...`
- Schemas: `TrendGapsResponse`, `TrendGapItem`, `CountryValidation`.
- Storage method: `signals_store.cross_country_gaps()`.

### UI

- Nueva página `/cazar/oportunidades` con matriz países × ideas:
  - 🟢 celda verde con número de signals = validada
  - 🔴 celda roja con ✱ = first-mover gap (oportunidad)
  - · celda gris = sin data
- Detalle de la "top oportunidad" con países validados + países gap +
  sample themes citados.
- Filtros: min_signals, min_feedback (controlable desde la UI).
- Link en sidebar: 🌎 Oportunidades.

### Tests (6 ✓)

- detecta validated_in_x_missing_in_y
- excluye_already_present_countries
- requires_validation_min_signals
- accepts_custom_countries
- rejects_invalid_args
- requires_auth

### Limitaciones conocidas

- Cluster por suggested_topic[:60] es heurístico — dos ideas con strings
  similares pero semánticamente distintas pueden agruparse falsamente. En
  M5 se reemplaza por embeddings cosine clustering.
- Normalización de países es limitada — solo soporta los abbreviation
  matches que listamos (US/USA/Estados Unidos, etc.). Países con
  variaciones ortográficas no contempladas no matchean.
- Señales con `country_focus: "LATAM"` (pan-regional) se IGNORAN para
  validación específica — esto es intencional: una idea "LATAM" no
  prueba que esté validada en un país concreto.

## M4.13 — Eventos / Ferias source kind (✅ shipped)

**Mapea a la cita #5 del audio.**

### Diseño

- Nuevo `source_kind = "events"` que delega en el `fetch_rss` existente.
- Justificación: la mayoría de plataformas de eventos (Lu.ma, Eventbrite,
  conferencias verticales) exponen feeds RSS estándar. Reciclar
  infraestructura existente vs construir un fetcher especializado da
  el mismo resultado con 1/10 del código.
- En M4.14+ podemos especializar con Eventbrite API si hay demanda.

### API

- `POST /api/v1/sources` ahora acepta `kind="events"`.
- Pattern regex actualizado en `SourceItem` y `SignalsDeleteByTypeRequest`.
- Aparece en filtros `/api/v1/signals?kind=events`.

### UI

- `/cazar/fuentes` dropdown del form de añadir tiene "Eventos / Ferias /
  Congresos (M4.13)" con placeholder `https://lu.ma/feed.xml`.
- `/cazar/senales` dropdown de filtro de fuente incluye "Eventos / Ferias".

### Tests (3 ✓)

- events_kind_accepted_by_sources_endpoint
- events_kind_filter_works_in_signals_endpoint
- events_kind_delete_by_source_kind

### Cómo usarlo

Suscribite a:
- `https://lu.ma/<categoria>.rss` — eventos de Lu.ma
- `https://www.eventbrite.com/rss/...` — Eventbrite
- Feeds RSS de TechCrunch Disrupt, Web Summit, Money 20/20, Bessemer
  events, etc.
- Calendario Google público como ICS+RSS via tools como `ical2rss`.

## M4.12 — Sleeper Companies Radar (🟡 roadmap, no implementado)

**Mapea a la cita #1 del audio.**

### Por qué NO se implementa ahora

Requiere fuentes de datos financieros públicos:
- **USA:** SEC EDGAR API (gratis, pero rate-limited y XML pesado)
- **EU/UK:** Companies House API (gratis pero auth + rate limits)
- **LATAM:** Datos no uniformes. En EC: Superintendencia de Compañías
  ofrece datos en PDFs scrape-ables. En MX/CO: cámaras de comercio
  públicas. No hay una API unificada.

Cada fuente requiere:
1. Parser específico (XML SEC vs JSON Companies House vs PDF SuperCias)
2. Normalizador de métricas (revenue, growth rate, EBITDA — formatos distintos)
3. Agente comparador #1 vs #2 vs #3 por industria
4. UI tipo screener financiero

**Estimación:** 2-3 semanas. ROI alto pero costo alto.

### Diseño propuesto para cuando se implemente

```
# Backend
- New source_kind = "financial_filings"
- New module: orchestrator/sources/sec_edgar.py
- New agent: sleeper_detector (compara métricas dentro de SIC code)
- Salida: signals con metadata {company, sic_code, peer_rank, growth_vs_leader}

# UI
- New page /cazar/empresas-radar
- Tabla: filtrar por sector, ordenar por (growth_rate - leader_growth_rate)
- Marcar "second-best with momentum"
```

### Mientras tanto

Como workaround manual: el founder puede añadir feeds RSS de
- Crunchbase (RSS premium pero algunos son gratis)
- TechCrunch /funding tag
- LATAM Ventures Newsletter
- Cualquier publicación de funding rounds

Y filtrar por content_type=blog + búsqueda por sector. No es óptimo pero
da signal hasta que tengamos SEC EDGAR integrado.

## M4.14 — Trending Products Radar / Dropshipping (🟡 roadmap)

**Mapea a las citas #3 y #4 del audio.**

### Por qué NO se implementa ahora

Las APIs de e-commerce trending son cerradas:
- **Aliexpress:** API privada (B2B partners). Scraping prohibido por TOS.
- **Amazon Best Sellers:** RSS por categoría existe pero deprecated; el
  scraping anti-bot está endurecido.
- **TikTok Shop:** API solo para sellers verificados.
- **Shopify trends:** Solo para tiendas con cuenta.

Alternativas legítimas:
- Google Trends API (gratis, pero limitado a queries en lugar de productos)
- Spocket / Trendsi (APIs de dropshipping paid)
- Pinterest Trends (API existe para business accounts)

### Diseño propuesto

```
# Backend
- New source_kind = "trending_products"
- Integrar Google Trends API → top rising queries por region/category
- Calculadora de margin: (precio_venta_usa - costo_aliexpress) × (1 - comisión)
- Signal output: {product_name, source_region, target_region,
                  margin_estimate_usd, shipping_complexity}

# UI
- New page /cazar/productos
- Tabla con product cards: imagen, márgen, regiones validadas, "duplica aquí"
- Click → workflow run con prompt pre-armado de ICP por región
```

**Estimación:** 1-2 semanas si nos quedamos con Google Trends.
3+ semanas si integramos Spocket o Trendsi paid.

### Mientras tanto

El founder puede subir CSVs con listas de productos a `/cazar/bitacora`
y los analizamos con el `link_analyzer` existente.

## M4.15 — Niche-en-gigante Detector (🟡 roadmap)

**Mapea a la cita #6 del audio.**

### Por qué NO se implementa ahora

Requiere:
- Base de datos de TAM (Total Addressable Market) por sector + región.
  Fuentes: Statista (caro), Gartner (muy caro), reportes McKinsey libres
  (no estructurados).
- Clustering semántico de "sub-verticales" — esto sí lo podríamos hacer
  con embeddings pero necesitamos un corpus grande de descripciones de
  empresas/productos. Crunchbase básico, Product Hunt, IndieHackers.
- LLM agent que razone "X es un sub-vertical infra-explotado de Y porque…"

### Diseño propuesto

```
# Phase 1 (cheap)
- Cuando idea_analyzer ejecuta, añadir un campo "parent_market" al output
  ("legaltech-autónomos" → parent: "legaltech")
- Clustering: agrupar por parent_market y contar #ideas en ese parent
- Si parent tiene 50+ ideas y un sub-vertical específico tiene <5 → flag
  como "niche in giant"

# Phase 2 (deep)
- Integrar reportes públicos de TAM por sector
- LLM agent compara: TAM_parent vs TAM_subvertical vs #competidores → score
```

**Estimación:** Phase 1: 1 semana. Phase 2: 3-4 semanas.

- "Governor" central que decide qué agente correr → NO. Mantener pipeline
  lineal con scopes exclusivos (M3 design).
- "Multi-agent debate" sin schema bien definido → fácil de hacer mal.
  Implementar como "ensemble vote" con esquema fijo (como ya hacemos en
  gate_decider) en lugar de free-form debate.

## Consequences

### Positivas

- M4.11 + M4.13 entregan **2 features inspiradas directamente en citas del
  founder** del mismo día, en una sesión.
- M4.11 abre la página `/cazar/oportunidades` que es **visualmente
  contundente** para una pitch deck — matriz países × ideas con celdas
  rojas marcando oportunidades concretas.
- M4.13 amplía el cazador a un dominio nuevo (eventos) **sin código
  nuevo de fetcher** — máximo ROI.
- ADR-022 documenta el roadmap completo de 6 features para que cualquier
  ingeniero (o futuro Claude) pueda retomar donde quedamos.

### Negativas / aceptadas

- M4.12, M4.14, M4.15 quedan como "intent to build" sin código. Riesgo de
  que se vuelvan vapor si no se ataca en M5/M6.
- El clustering heurístico de M4.11 va a generar falsos positivos en casos
  edge (suggested_topic vacío o muy genérico). Necesita refinamiento con
  embeddings cuando lleguen los volúmenes.
- El kind `events` reusa `fetch_rss` — si Lu.ma o Eventbrite cambian su
  estructura RSS, rompe silenciosamente. Mitigación: monitor de scan health
  ya existe (M3.x).

## Alternatives considered

| Alternativa | Por qué NO |
|---|---|
| Construir las 6 features en un solo sprint | Imposible en una sesión. 3 requieren APIs externas no triviales. |
| Hacer un agente único "audio2roadmap" con LLM | Sobre-ingeniería. Mejor escribir el ADR a mano. |
| Implementar M4.11 con embeddings vectoriales directamente | Embeddings ya existen (M4.1) pero el clustering de M4.1 está en stale state. Heurístico con suggested_topic resuelve el 80% del caso a 1/10 del costo. |
| Crear un nuevo agente sleeper_detector con prompt creativo sin datos | Fundamentalmente roto. Sin datos financieros públicos reales, el LLM alucina. |

- Frontend: páginas nuevas sin afectar las existentes

## Lessons learned

1. **Cuando el founder graba un audio sobre su modelo de negocio, hay
   roadmap fresh para 3-6 semanas.** Convertir audio → ADR estructurado
   es 1 hora de trabajo con altísimo ROI estratégico.
2. **Cluster heurístico con suggested_topic[:60]** resuelve el 80% del
   caso a 1/10 del costo de embeddings. Embeddings se justifica cuando
   tengamos >10k signals y queramos disambiguation de polisemia.
3. **No fingir features que requieren APIs externas.** ADR explícito de
   "esto NO está implementado y por qué" vale más que un endpoint stub
   que retorna mock data.
