# ADR-012: Expand Cazador with YouTube + Bluesky + Telegram + Trend Score

**Date:** 2026-05-27
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M3.1 — circles-ai.ai

---

## Context

ADR-011 established the autonomous hunter with 6 source kinds (URL, RSS, HN,
Reddit, GitHub Trending, Product Hunt). User feedback after first real test
("no signals from my Instagram URLs") exposed two gaps:

1. **Coverage gap** — the configured sources didn't cover where LATAM founders
   actually post: YouTube creators discussing pain points, Bluesky's new
   tech community, Telegram channels (LATAM uses Telegram much more than X).

2. **No temporal awareness** — a signal that appears once is treated the
   same as one that reappears across 5 days. The hunter has no notion of
   "trending" vs "isolated mention".

Also: the mock scanner used to refuse single-item batches, which made manual
testing with URL-kind sources show "0 signals" — confusing UX.

## Decision

Three additive expansions, all backwards-compatible:

### 1. Three new source kinds (all free, all stdlib-only)

| Kind | Auth | Endpoint | Notes |
|---|---|---|---|
| `youtube` | none | `/feeds/videos.xml?channel_id=UCxxx` | Per-channel RSS, no API key. Helper resolves @handle / /channel/ / /c/ → channel_id. |
| `bluesky` | none | `https://public.api.bsky.app/xrpc/app.bsky.feed.searchPosts` | Public AT Protocol XRPC, no token. |
| `telegram` | none | `https://t.me/s/<channel>` (HTML scrape) | Public channels only. Private channels return empty silently. |

Explicitly **not** in this sprint:
- X/Twitter ($100/mo Basic plan)
- LinkedIn (partner-only API)
- TikTok (research-only API)
- Instagram (Graph API exposes only your own posts, useless for hunting)

### 2. Trend score on signals

Added column `signals.trend_score REAL NOT NULL DEFAULT 0` via additive
migration (`ALTER TABLE ... ADD COLUMN`). Backfill on insert:

> `+1` for each prior signal in the last 7 days whose theme shares a 5+
> char keyword with the new signal. Capped at 5.0.

Signals are now sorted by `trend_score DESC, score DESC, created_at DESC` —
so reappearing themes float to the top of the review queue regardless of
which day they happened to score highest.

### 3. Mock scanner accepts single-item batches

`SourceScannerAgent._mock_scan` previously required 2+ items per batch,
which produced 0 signals for any URL-kind source (URL always yields 1
item). Now:

- 0 items → 0 signals
- All-empty-body items → 0 signals (Instagram-like URLs)
- 1 item with content → 1 signal, score 0.60 (single-source confidence)
- 2+ items → 1 signal, score 0.72 (cross-source pattern)

Live (non-mock) scanner behavior unchanged — its scoring already handled
single-item batches because the LLM judges quality, not item count.

## Why these and not (X)

| Alternative | Why not |
|---|---|
| **X/Twitter API Basic ($100/mo)** | Pure cost. Defer until paid-product validates revenue. |
| **YouTube Data API v3 (paid quota)** | The free per-channel RSS covers 80% of value at $0. Only need the paid API for search-across-all-of-YouTube which we don't need. |
| **TikTok Research API** | Academic-only. Not realistic. |
| **LinkedIn API** | Partner application required, basically impossible at our scale. |
| **Instagram Graph API** | Returns only the authenticated user's own content. Not for hunting. Tested — user found this out the hard way. |
| **Nitter (Twitter scraper)** | Public Nitter instances are unreliable and constantly fail. Skip for now. |
| **YouTube Data API search** | Free quota of 100 searches/day is tempting but the per-channel RSS gives better signal-to-noise: founders who run a channel post pain points, random video search returns spam. |
| **Trend detection via NLP clustering** | Embedding-based dedup would be cleaner but needs sentence-transformers or external API. Word-overlap is good enough until we have ≥500 signals to need the upgrade. |

## Cost & risk

- All three new kinds are **$0 incremental** (no LLM calls during fetch, no
  paid APIs).
- Telegram scraping is brittle (HTML structure may change). Mitigation:
  `fetch_telegram` returns `[]` silently on parse failure; logged warning.
- Bluesky public API may add auth requirements in the future. Mitigation:
  ADR-012 follow-up if that happens; until then, no action.
- YouTube channel resolution via scraping may break if YouTube changes
  their channelId embedding. Mitigation: `_resolve_youtube_to_rss` returns
  `None` on failure, source is skipped with a warning logged.

## What this does NOT change

- The 5-step linear chassis (workflow architecture).
- The 6 active agents (no new agent added in this ADR).
- The ensemble model (still Claude+OpenAI+Gemini only on `gate_decider`).
- Authentication / authorization (closed-beta allowlist still applies to
  every endpoint that creates sources or scans).

## Trigger to revisit

- > 100 signals from YouTube + Bluesky + Telegram → harvest data to see
  which kind has the best signal:noise ratio, weight accordingly
- Trend score consistently 0 across N=50 signals → the keyword-overlap
  heuristic is too strict; upgrade to embeddings
- User asks for X/Twitter → revisit ADR with revenue-justification
