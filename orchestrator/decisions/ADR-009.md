# ADR-009: Layered Anti-Bot Defenses (R26)

**Date:** 2026-05-26
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M1.6 — circles-ai.ai

---

## Context

Two endpoints now face the public internet:

1. **`POST /api/v1/leads`** — lead capture from `/f/[slug]` landing pages.
   Form submissions are cheap individually but a single spam burst could fill
   the founder's review queue with garbage, drown out real signals, and bias
   the gate decision toward false PASS verdicts.

2. **`POST /api/v1/gate/run`** — runs the full 6-agent EvidenceGateWorkflow.
   Each call costs ~$0.06 in Anthropic + OpenAI + Gemini tokens. An attacker
   discovering the unauthenticated endpoint and looping it would drain the
   API budget in hours.

A single defense (e.g. only rate-limiting) leaves a known gap. CAPTCHAs
alone hurt conversion. We need **layered** defenses where each layer is cheap
to add and each layer covers a different attack profile.

## Decision

Five composable defense layers in `orchestrator/core/anti_bot.py`, each
independently opt-in or always-on:

| # | Layer | Endpoint | Activation | Bot profile covered |
|---|---|---|---|---|
| 1 | Per-IP burst limit | both | always-on | Naive flooders |
| 2 | Per-IP daily quota | `/gate/run` | always-on | Slow drip attacks |
| 3 | Honeypot field + dwell time | `/leads` | always-on | Generic form-fillers |
| 4 | Cloudflare Turnstile | `/leads` | opt-in (`TURNSTILE_SECRET_KEY`) | Headless browsers |
| 5 | Disposable-email blocklist | `/leads` | always-on | Throw-away accounts |
| 6 | Shared-secret header | `/gate/run` | opt-in (`GATE_RUN_SECRET`) | Private deployments |

**Tiered quotas** so cheap public endpoints don't share quota with
expensive LLM endpoints:

```
public_form:    5 per minute,   30 per day per IP
expensive_llm:  2 per 5 min,    10 per day per IP
```

All quotas overridable via env vars (`RL_FORM_BURST_MAX`, etc.) for ops
emergencies.

## Why these layers and not (X)

| Alternative | Why not |
|---|---|
| **reCAPTCHA v3** | Privacy concerns (LOPDP Ecuador), Google tracking, UX friction. Turnstile is free + privacy-respecting + invisible. |
| **JWT auth on all endpoints** | Lead forms must be public. We can't ask landing visitors to log in. |
| **WAF / Cloudflare full Bot Management** | $200/month minimum. Premature for M1; the layers above cover 95% of attacks. |
| **Single static API key for /gate/run** | Same risk as no auth if leaked. Shared-secret header is just-as-strong AND can be rotated trivially via env var. |
| **Database-backed rate limit (Redis)** | Adds dependency. In-process is fine for M1 (1 backend instance). M3+ moves to Redis when we scale horizontally. |

## Honeypot strategy: silent-accept

When the `company_website` field is filled (bot trip), we return **HTTP 201
with `accepted=true`** and discard the lead silently. The bot doesn't learn it
was caught — so it doesn't iterate to bypass us. Real humans never see the
field (it's positioned at `left: -10000px` and `tabindex=-1`).

## Cost & latency

- Layers 1, 2, 3, 5: **0 ms** added (in-process dict lookups, regex)
- Layer 4 (Turnstile): **+50-150 ms** when enabled (Cloudflare verify call)
- Layer 6: **0 ms** added (string compare)

## What this does NOT do

- Does NOT replace HTTPS, security headers, CORS, or CSP — those are R20.
- Does NOT protect against application-layer DoS (a single legitimate user
  refreshing the gate endpoint manually). For that we have the daily quota.
- Does NOT block sophisticated, low-volume targeted scrapers. Those would
  bypass layers 1+2 (low rate), 3 (real browser), 4 (humans solve Turnstile),
  5 (real email). For those we'd add behavioral analysis in M3+.

## Operational runbook

If we get spammed:

1. **Burst spike**: lower `RL_FORM_BURST_MAX` env var (no deploy needed)
2. **Daily quota hit**: lower `RL_FORM_DAILY_MAX`
3. **Turnstile signal degrading**: rotate Turnstile site/secret keys, no code change
4. **Suspected bypass of /gate/run**: set `GATE_RUN_SECRET` env var → all clients
   must include `X-Gate-Secret` header. Old clients get 401 until updated.

## Trigger to revisit

- > 5% of legitimate-looking submissions get blocked → loosen
- < 1% submissions blocked over 30 days → tighten or remove layers that aren't earning their keep
- Single attacker bypasses everything → add behavioral analysis (mouse paths, scroll, etc.)
