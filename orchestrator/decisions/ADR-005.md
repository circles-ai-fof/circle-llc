# ADR-005: Defer "Claude Code vs Codex" Auto-Selection

**Date:** 2026-05-26
**Status:** Accepted (deferred decision)
**Deciders:** Cristian Molina (Circle LLC)
**Context:** Sprint M1 — circles-ai.ai Factory of Factories

---

## Context

User asked whether the orchestrator should auto-decide between **Claude Code** and **OpenAI Codex** for the build phase of each idea that passes the evidence-gate.

The intent: route some ideas to Claude Code (better for refactors, multi-file edits, system design) and others to Codex (faster for isolated functions, narrow lambda-style transforms) based on signal from the idea's vertical, complexity, and codebase shape.

## Why we deferred

We do not yet have the data to make this decision well, and shipping the wrong heuristic into the chassis would propagate noise into every downstream gate.

The specific gaps are:

1. **Zero ground-truth pairs.** We have not built a single idea with both tools side-by-side. Any router we write today is opinion, not measurement.
2. **No comparable cost surface.** Codex pricing, tooling, and integration model in 2026 changed (OpenAI Agent Builder + Codex CLI 2.x); we need fresh benchmarks, not a recalled 2024 mental model.
3. **Outcome DB is inactive (R10).** The right place to log "tool X built feature Y in Z hours with Q rework" is the Outcome DB, and that is intentionally disabled until N≥3 factories complete the gate.
4. **Reganti Cap 10 §10.3 applies twice:** "stay at the simplest level that handles 90% of your cases." Today, 100% of our cases are handled by a single tool. Adding a router with zero data violates this principle.
5. **Anti-pattern recognition.** Building a "Governor" agent that dynamically routes downstream tools is exactly the architecture Reganti Cap 9 §9.4 warns against without an explicit override.

## Decision

**Defer to M3.** The router will be designed only when ALL of the following are true:

- N ≥ 10 ideas have been built (any tool)
- Outcome DB is active and has ≥ 50 logged build outcomes with `tool_used`, `hours_spent`, `rework_count`, `success_metric`
- A new ADR is written documenting the empirical signal (not opinion) that justifies a router
- The router is implemented as a single deterministic Python function with a code-based eval suite (Cap 7)

Until then, the build tool will be **operator choice**, recorded as a free-text field in the run's metadata. This gives us the data we need to make the decision later, without committing the chassis to a guess.

## What we are NOT doing (explicitly)

- No `tool_router_agent` in `orchestrator/agents/`
- No `build_tool_selector` step in `EvidenceGateWorkflow`
- No conditional branch in the workflow based on `idea.vertical_category` or any other heuristic
- No deferred config flag (`AUTO_SELECT_BUILD_TOOL`) — feature does not exist

## Consequences

**Positive**
- Chassis stays a 5-step linear workflow (no dynamic routing — preserves Reganti workflow-first principle)
- Build-phase metadata captures the missing data passively, so the decision can be revisited with evidence
- No code or tests written that would need to be removed when the empirical answer differs from our 2026-05 intuition

**Negative**
- Operators must pick a build tool manually for the first 10 factories (low cost; 2 founders today)
- If a clear pattern emerges anyway (e.g., "all fintech ideas need Claude Code") we will recognize it from logs, not from a router

## Trigger to revisit this ADR

Either:
- The Outcome DB has logged ≥50 build outcomes with measured `tool_used` and rework metrics (default trigger), OR
- A founder writes a one-page memo arguing a specific routing heuristic with N≥10 supporting examples (override trigger)
