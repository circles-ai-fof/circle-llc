# ADR-027: M6.0 — MultiAgentConsensus (último agente del audio del founder)

**Date:** 2026-05-30
**Status:** Accepted — agente experimental shipeado con 10/30 golden cases
**Deciders:** Cristian Molina
**Context:** Última cita del audio del founder pendiente de cobertura:
*"vamos a tener unas estructuras de agentes con razonamientos tan avanzados
que van a poder analizar entre ellos, para hacer los insights, para poder
tomar decisiones"*. Este ADR materializa esa idea con un agente sintetizador
schema-fijo, compatible con R11 (no governor).

---

## Context

R11 del rulebook prohibe explícitamente "agentes que decidan dinámicamente
qué agente llamar" (Governor pattern). Pero el founder pide razonamiento
INTER-agente. Esa tensión se resuelve con un patrón conocido en multi-agent:

**El agente NO es un router de otros agentes — recibe outputs ya producidos
y los sintetiza con schema FIJO.**

La responsabilidad de QUÉ agentes invocar antes (y en qué orden) pertenece
al endpoint orquestador o al frontend del founder, no al agente sintetizador.

## Decision

### Diseño del MultiAgentConsensus

**Input shape** (validado por Pydantic):
```python
{
  "decision_question": str (1-500 chars),
  "perspectives": [           # 1-10 items
    {"source": str (1-100), "text": str (1-2000)},
    ...
  ]
}
```

**Output shape** (también fijo):
```python
{
  "agreement_score": float (0.0-1.0),
  "consensus_view": str,            # área común de acuerdo
  "dissenting_views": List[str],    # opiniones que se salen del consensus
  "key_tradeoffs": List[str],       # ejes de decisión con divergencia
  "final_recommendation": str,      # con caveat si dispersion alta
  "confidence": float (0.0-0.85),   # nunca > 0.85 (incertidumbre intrínseca)
  "reasoning": str
}
```

### R11 compliance — POR QUÉ no es Governor

Un Governor sería:
- Recibe una pregunta abierta
- Decide qué agentes invocar (Trend? Niche? Sleeper?)
- Coordina sus respuestas en tiempo real
- Devuelve una decisión

Lo que MultiAgentConsensus hace en cambio:
- Recibe perspectives YA PRODUCIDAS (caller las generó antes)
- NUNCA invoca otros agentes — el campo `_client` solo se usa para el LLM
  judge call sobre el texto provisto
- Schema de input/output FIJO — sin "elegir" qué dimensión cubrir
- El caller decide qué perspectives meter (frontend conoce qué agentes
  invocó)

Es un puro sintetizador, no un orquestador.

### Casos de uso reales del founder

1. **Founder no confía en TrendGapAnalyzer solo:**
   ```
   Caller: frontend del dashboard
   1. Invoca POST /trend-gaps/analyze → obtiene plan A
   2. Invoca POST /trend-gaps/analyze con threshold distinto → plan B
   3. Founder escribe su gut feeling como texto
   4. POST /consensus/analyze con las 3 perspectives + decision_question
   ```

2. **Decisión de promote/skip con segundo opinion:**
   ```
   1. /signals/{id}/analyze → recomendación del idea_analyzer
   2. /trend-gaps/analyze → contexto cross-country
   3. /consensus/analyze → ¿estas dos perspectives concuerdan?
   ```

3. **Multi-fábrica consensus (M6+ cuando haya >1 fábrica):**
   ```
   Mismo dilema en 3 fábricas distintas → 3 perspectives → consensus
   ```

### Heurística mock (sin LLM)

Para CI y dev sin API key:

- **agreement_score**: Jaccard sobre conjuntos de palabras (len>3) de
  cada perspective. Normalizado a [0.15, 0.95] estirando con factor ×2.5.
- **consensus_view**: la perspective con más overlap con el "common set"
  (intersección de todas) — la "más central".
- **dissenting_views**: perspectives cuyo overlap con common es < 30%
  del tamaño del common.
- **confidence**: 0.4 + agreement × 0.5, cap 0.85.
- **final_recommendation**: gradiente según agreement (alto → procede,
  medio → cautela, bajo → no actuar).

Esto es 30 líneas de Python y produce outputs sensatos para los 10 cases.

### Confianza máxima 0.85 (regla del prompt + mock)

Decisiones de alto stakes nunca tienen 0.9+ de confianza real. El cap
es un guardrail anti-overconfidence — el agente NO debe convencer al
founder de que "el consenso es 95% claro" cuando hay perspectives
divergentes.

## Consequences

### Positivas

- **Última cita del audio cubierta.** Mapeo completo:
  - "first-mover ahí" → trend_gap_analyzer ✅
  - "migajas de gigantes" → niche_scout ✅
  - "ferias y congresos" → event_relevance_scorer ✅
  - "segundo de a bordo" → sleeper_company_detector ✅
  - "86 centavos / 19.99" → product_arbitrage_evaluator ✅
  - "agentes razonando entre ellos" → multi_agent_consensus ✅ (este ADR)
- **13 agentes** (vs 7 al inicio): 7 workflow + 5 active(on-demand) +
  1 experimental. Sigue lejos del target de 40 del founder pero el patrón
  para llegar está demostrado 6 veces.
- **R11 preservado** — el agente es un sintetizador, no un router.
  Cualquiera que mire SCOPES.md ve el "NO invoca otros agentes" explícito.

### Negativas / aceptadas

- **10 cases es threshold mínimo.** Promoción a active(on-demand)
  requiere 20 más (mismo patrón que M5.0→M5.1 y M5.2-5→M5.8-11). Por
  ahora `EXPERIMENTAL = True`.
- **Heurística mock simple.** El Jaccard no captura semántica — dos
  perspectives con palabras distintas pero significado idéntico van a
  reportar bajo agreement. Mitigación: prompt real con LLM hace ese
  trabajo. El mock es para CI/dev solamente.
- **Cap de 0.85 confidence** podría frustrar al founder en casos
  donde el consenso es realmente sólido. Trade-off aceptado:
  anti-overconfidence > expressiveness.

## Alternatives considered

| Alternativa | Por qué NO |
|---|---|
| Hacer un Governor real que invoque agentes dinámicamente | Viola R11 explícitamente. La regla existe por razones (caps 14-15 del Handbook). |
| Implementar el agente con N llamadas al LLM (ensemble vote multi-modelo) | Más caro ($0.05+ por análisis vs $0.012 con single call). El gate_decider ya hace eso para verdict — no necesitamos duplicar. |
| Saltarse el agente y dejar al founder hacer la síntesis a mano | Pierde el valor del audio del founder. El founder pidió explícitamente "razonamientos entre agentes". |
| Activar directo como `active` (saltar experimental) | Viola R12. Mismo patrón que M5.x — pasa por experimental con 10 cases, luego promoción. |

## Tests

10 golden cases (mock_mode, para CI):
1. Flow básico (2 perspectives con overlap textual)
2. Single perspective → agreement=1.0 trivial
3. Perspectives idénticas → agreement alto
4. Perspectives contradictorias → agreement bajo
5. confidence ∈ [0, 0.85]
6. Auth required (401)
7. Empty perspectives → 422
8. > 10 perspectives → 422
9. Empty decision_question → 422
10. mock_mode=True + cost=0.0

Total: 721 tests verdes (vs 711 antes).

## Migration

Aditiva. Sin breaking changes. Endpoint nuevo, schemas nuevos. Frontend
no requiere cambios hoy — la integración del botón "🤖 Sintetizar"
en, por ejemplo, /cazar/oportunidades queda para M6.0b cuando el founder
realmente lo demande.

## Roadmap del agente

- **M6.0** ✅ (este ADR): experimental con 10/30

  bajo en mock indica que vale la pena el gasto LLM real
