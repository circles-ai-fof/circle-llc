# ADR-006: Tool-Use in idea_enricher (web_search) — opt-in

**Date:** 2026-05-26
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M1.6 — circles-ai.ai

---

## Context

The original `idea_enricher` (Sprint M1.5) sharpens vague ideas using ONLY the LLM's parametric knowledge of LATAM markets. This works for high-frequency patterns the model has memorized, but produces hallucinated numbers for niche verticals ("60% de PYMEs Ecuador esperan 90 días" — plausible-sounding but unverifiable).

User asked: when ideas are vague, AIs should "ask for more information" — not just hallucinate sharper-sounding outputs.

Reganti Cap 12 §12.4 endorses tool-use loops for this exact case: *"An agent with retrieval is stronger than an agent alone — when grounding matters, the tool call is cheaper than the hallucination."*

## Decision

Add **opt-in `web_search` tool-use** to `idea_enricher` using Anthropic's native server-side web search (`web_search_20250305`).

Activation rules:
- **Enabled when**: `IDEA_ENRICHER_RESEARCH=true` (env var) AND `ANTHROPIC_API_KEY` set AND not in `mock_mode`
- **Disabled when**: env var absent, mock mode, or API key missing → enricher uses parametric path (current behaviour)
- **Hard cap**: `max_uses=3` per enrichment call → bounds cost/latency
- **Cost cap**: each web_search adds ~$0.01 + Anthropic's tool surcharge → at most $0.05 per enrichment
- **Falls back gracefully**: if web_search returns errors, enricher continues with parametric output

## Why this and not (X)

| Alternative | Why not |
|---|---|
| **Always-on web search** | Adds latency + cost to ideas that don't need grounding (already-specific inputs). Specificity score < 3.5 is a cheap gate; only fire web search after the gate trips. |
| **Multi-LLM debate to ground claims** | Different models share the same training data biases. They will agree on the same hallucination. Real web data is the only fix. |
| **External search API (SerpAPI/Brave)** | Adds dependency + key management. Anthropic's native tool has no extra account, and the search results are inline in the model's reasoning loop. |
| **Build our own retrieval over a curated LATAM dataset** | M3+ territory. Premature without N≥3 factories. |

## What this changes

- `orchestrator/agents/idea_enricher.py`: new internal helper `_enrich_with_research()` that calls Claude with `tools=[{"type": "web_search_20250305", ...}]` when activation conditions hold.
- `orchestrator/agents/SCOPES.md`: idea_enricher scope updated to include "may call web_search to ground claims".
- `orchestrator/rulebook.md`: R23 added — *Research tool-use limited to idea_enricher; gate_decider stays one-shot per ADR-005*.
- `R13` budget tracker: each web_search call counts as one tool step against the trajectory cap.

## What this does NOT change

- The 6-step linear chassis (hunter → enricher → maturer → market_validator → landing → gate). No new agents.
- `gate_decider` stays one-shot (no tool use in classification).
- Mock mode still produces deterministic output (web_search never invoked under `mock_mode=True`).

## Consequences

**Positive**
- Vague-idea inputs get grounded in real market data (competitors, regulations, size estimates) instead of plausible-sounding fabrications.
- Logs each web_search call so we can later audit "did the research actually change the output?" and shut it off if not.

**Negative**
- Cost per vague-idea enrichment: +$0.01–0.05 (acceptable).
- Latency per enrichment: +5–15s when web_search fires (only on vague inputs).
- Adds Anthropic-specific dependency surface (Bedrock/Vertex relay would need to support the tool).

## Trigger to revisit

Either:
- Web search fires on >50% of enrichments → too aggressive; tighten gate.
- Web search fires on <5% of enrichments → not earning its keep; reconsider always-on grounding instead.
