# ADR-017: Auditoría integral y endurecimiento (M3.12)

**Date:** 2026-05-28
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M3.12 — auditoría integral solicitada por el founder

---

## Context

Tras 11 sprints de iteración rápida (M3.0 → M3.11) el founder pidió:
> "Valida la aplicación integral incluido la arquitectura y mejórala si es necesario."

Hice una auditoría sistemática del proyecto en 8 dimensiones:
1. Estructura y tamaño de archivos
2. Secretos y .gitignore
3. Autenticación y autorización
4. SQL injection y prepared statements
5. Frontend XSS / hardcoded URLs / console leaks
6. Persistencia y race conditions
7. Índices y performance
8. CI/CD y deploy

## Hallazgos

### 🔴 CRÍTICO: 2 endpoints sin autenticación

**Issue**: `GET /api/v1/gate/pending-review` y `POST /api/v1/gate/runs/{id}/human-override`
sólo tenían `_check_rate_limit` pero **no `_require_user`**.

Impacto:
- Cualquier persona con conocimiento del run_id (UUID, 36 chars) podía
  **modificar el verdict** de una decisión del workflow.
- `pending-review` listaba ideas estratégicas en proceso de evaluación
  (filtración de inteligencia competitiva).

Esto fue un olvido — los endpoints de hunter (`/signals/*`, `/sources/*`,
`/stats`, etc.) sí tenían `_require_user`, pero los de review se escribieron
antes de la auth allowlist (ADR-010) y nunca se actualizaron.

**Fix aplicado** (este ADR):
- `list_pending_review(request)` ahora llama `_require_user(request)`
- `post_human_override(...)` también
- Frontend `app/revision/page.tsx`: cambiado `fetch()` → `authFetch()`
  para incluir el `Authorization: Bearer ...` header
- Tests `test_human_review.py` actualizados con fixture `auth`
- 2 tests nuevos: `test_pending_review_requires_auth`,
  `test_human_override_requires_auth` (verifican que sin token devuelve 401)

### 🟡 IMPORTANTE: defaults `:8000` vs `:8002`

**Issue**: 4 archivos del frontend tenían:
```ts
const API = process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:8000";
```

Pero el orchestrator se documenta corriendo en `:8002` (DEPLOY.md, .env.local).
Si el founder ejecuta sin `.env.local`, todas las requests van a `:8000`
(que en su máquina es otro proyecto). Network errors silenciosos.

**Fix**: bulk replace `:8000` → `:8002` en:
- `app/cazar/page.tsx`
- `app/leads/page.tsx`
- `app/revision/page.tsx`
- `lib/auth.ts`

`app/configuracion/page.tsx` mantiene `:8000` pero solo en texto de display
(no fetch), así que lo dejé.

## Decision

Mantener la arquitectura actual con los siguientes endurecimientos:

1. **Auth obligatoria en todo endpoint que escriba o lea estado privado**.
   `_require_user(request)` debe estar como primera línea del handler.
   Excepciones documentadas: `/health`, `/agents`, `/diagnostic`,
   `/leads/capture` (público para landings), `/auth/login`.

2. **Defaults consistentes en código y docs**.
   Backend local = `:8002`. Producción = Railway URL via `NEXT_PUBLIC_API_URL`.

3. **CI incluye dashboard build**. Cada push al master pasa por TypeScript
   + Next build además de pytest. Previene "el código compila pero el deploy
   falla".

4. **Refactors invasivos diferidos a M4**.
   - `senales/page.tsx` (1085 LOC) → 5 componentes separados
   - `api.py` (2226 LOC) → routers por feature (hunter, gate, leads, auth, meta)
   Ambos son alto-riesgo y bajo-impacto inmediato.

## Why these choices

| Decisión | Por qué |
|---|---|
| Auth en review endpoints en vez de mantenerlos públicos | Modifican decisiones críticas del workflow — el rate-limit no es suficiente protección. |
| `_require_user` (closed beta) en vez de `_is_admin` (X-Gate-Secret) | El founder ya está en la closed beta. Los emails allowlisted son los únicos con permiso. Más simple que mantener dos esquemas de auth para el mismo recurso. |
| No purgar mocks automáticamente al startup | El founder podría querer mantenerlas para debug. Mejor exponer el botón "Borrar mocks" y dejarlo decidir. |
| No refactor de `api.py` en este sprint | 2226 LOC y 50+ endpoints. Un error tipográfico en un decorator pasaría tests y rompería prod. Reganti Cap 13: "delay large refactors until you have ≥2 weeks of stable behavior to compare against". |
| CI dashboard-build aunque tarda 3 min | El single failure mode que más cuesta tiempo es "tests pasan, deploy revienta". Vale 3 min de CI evitarlo. |

## What this does NOT do

- **No agrega 2FA** — el closed-beta allowlist + email+token simple es suficiente para 3 usuarios. Cuando se abra el beta a 50+, considerar 2FA o passwordless.
- **No agrega rate-limit por user** — sigue siendo por IP (R26). Razonable hasta que aparezcan usuarios abusivos.
- **No persiste runs históricos completos** — sólo el último `_runs` dict. Outcome DB (M4) lo arregla.
- **No habilita FTS5** — search server-side con LIKE basta hasta 5k señales (ADR-016).
- **No refactoriza componentes UI grandes** — diferido a M4.
- **No añade tests E2E (Playwright)** — diferido. Los smoke tests con curl + test client cubren los flujos críticos.

## Cost impact

Cero — todos los cambios son fixes de seguridad/DX, no cambian behavior LLM
ni añaden llamadas. Costo de operación: igual.

## Tests

- **+2 tests nuevos** (483 verdes, era 481):
  - `test_pending_review_requires_auth` → 401 sin token
  - `test_human_override_requires_auth` → 401 sin token
- **7 tests existentes** actualizados para pasar `auth` fixture
- **CI dashboard-build** ahora protege contra regresiones de TypeScript / Next build

## Trigger to revisit

- **Cualquier nuevo endpoint que escriba a `_runs` o `signals_store`** sin `_require_user` → bloquear en code review
- **Más de 5k señales en BD** → migrar search a FTS5 (ADR-016 documenta la migración)
- **`senales/page.tsx` supera 1500 LOC** → forzar refactor a componentes
- **Reportes de bug "se cae el dashboard"** → revisar el dev server cache (`.next` corruption)
- **Founder reporta lentitud con >100 señales** → añadir cache de `/stats` o virtualización en la lista
