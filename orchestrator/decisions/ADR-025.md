# ADR-025: M5.2–M5.5 — 4 agentes experimentales sobre los outputs M4.x

**Date:** 2026-05-30
**Status:** Accepted — los 4 agentes shipeados como experimental con 10 cases cada uno
**Deciders:** Cristian Molina
**Context:** Tras M5.1 (TrendGapAnalyzer ACTIVE), el founder pidió "si avanza
todo". Este ADR shipea los 4 agentes restantes del roadmap M5 (M5.2–M5.5)
siguiendo el patrón establecido en ADR-023 y ADR-024.

## Decision

### Los 4 agentes shipean simultáneamente como `experimental`

Cumple el threshold mínimo del estado experimental (ADR-023: ≥10 golden
cases). Cada uno tiene:

- **BaseAgent subclass** en `orchestrator/agents/<name>.py`
- **System prompt** con principios + reglas de salida JSON
- **Mock-aware** — sin API key produce placeholders "[Demo]" en español
- **Endpoint** POST `/api/v1/<scope>/...` con auth + validation
- **10 golden cases** en `tests/api/test_hunter.py`
- **Row en SCOPES.md** con counter `10/30`
- **`EXPERIMENTAL = True`** en la clase + `AGENT_VERSION = "0.1.0"`

### M5.2 — NicheScout

**Input:** NicheOpportunity (parent_market, parent_size, leader_niche,
underexplored_niches).
**Output:** plan de entrada al sub-niche más prometedor —
target_subniche, entry_thesis, competitive_advantage, MVP, validation_metrics,
estimated_capture_pct, key_risks, confidence.
**Mock heurística:** elige el primer underexplored (menor signal density),
confidence sube con parent_size (5/10/20 → 0.48/0.60/0.72).
**Endpoint:** `POST /api/v1/niche-opportunities/analyze`.

### M5.3 — EventRelevanceScorer

**Input:** event_title + event_description + opcional industry_focus.
**Output:** relevance_score, expected_attendees_profile, networking_value,
learning_value, estimated_cost_usd, expected_roi, recommendation
(go/skip/send_someone_else), preparation_topics.
**Mock heurística:** keywords GO ("fintech", "ai", "y combinator", "summit",
"disrupt", "web summit") → score 0.78 + go. Keywords SEND ("latam",
"regional", "expo", "feria") → 0.58 + send_someone_else. industry_focus
boost +0.15.
**Endpoint:** `POST /api/v1/events/score`.

### M5.4 — SleeperCompanyDetector

**Input:** lista de empresas con sus filings recientes.
**Output:** sector_summary, leader_candidate, sleeper_candidates,
comparison_signals, threat_assessment, investment_thesis, confidence.
**Mock heurística:** primer empresa = líder asumido, siguientes = sleeper
candidates con tesis basada en cadence de filings. Confidence 0.0 con 1
empresa (no comparable), 0.45 con 2, 0.62 con 3+.
**Endpoint:** `POST /api/v1/sleeper-companies/detect`.

### M5.5 — ProductArbitrageEvaluator

**Input:** trending_query (con prefijo `[GEO]`) + opcional target_geo,
source_cost_usd, target_price_usd.
**Output:** is_physical_product, product_category, source/target_region,
margin_estimate_pct, shipping_complexity, time_to_test_weeks, key_risks,
recommendation (test/skip/deepdive), confidence.
**Mock heurística:** PRODUCT_KEYWORDS (cargador, audífonos, anillo, gadget,
...) → producto físico, recomienda test/deepdive según si hay precios.
NON_PRODUCT_KEYWORDS (weather, elections, news, ...) → skip.
**Endpoint:** `POST /api/v1/arbitrage/evaluate`.

## Consequences

### Positivas

- **Patrón ahora demostrado 5 veces.** Cualquier nuevo agente sigue
  este molde: BaseAgent + system prompt + mock heurística + 10 cases
  → experimental, +20 cases adicionales → active(on-demand).
- **El founder puede invocar los 4 desde el dashboard** en próximos
  sprints — las UIs ya tienen las páginas base (/cazar/oportunidades,
  /cazar/nichos, /cazar/senales). Solo falta el botón "🤖 Analizar
  con IA" en cada panel.
- **Cobertura del audio:** ahora hay agente LLM para CADA cita
  estratégica del founder:
  - "first-mover ahí" → trend_gap_analyzer (M5.0/M5.1)
  - "migajas de gigantes" → niche_scout (M5.2)
  - "en qué ferias hay que estar" → event_relevance_scorer (M5.3)
  - "segundo de a bordo... financial pública" → sleeper_company_detector (M5.4)
  - "86 centavos / 19.99" → product_arbitrage_evaluator (M5.5)
- **613 tests verdes** — 40 nuevos esta sesión sin regresión.

### Negativas / aceptadas

- 4 agentes simultáneos en `experimental` significa que el founder podría
  invocarlos antes de que tengamos métricas reales de quality.
  Mitigación: response incluye `mock_mode: bool` flag — UI puede mostrar
  badge "DEMO" cuando aplica.
- Los mock heurísticos son simples — keyword matching, no embeddings.
  Funcionan para CI pero pueden ser inconsistentes con queries borderline.
  En producción con API key real los prompts hacen el trabajo pesado.
- No se hicieron UI integrations todavía. Cada agente tiene endpoint funcional
  pero el botón "🤖 Analizar con IA" solo está en /cazar/oportunidades
  (M5.0). UI para los otros 3 queda para sprints siguientes.

## Alternatives considered

| Alternativa | Por qué NO |
|---|---|
| Shipear los 4 directamente como active (30 cases c/u) | 120 nuevos tests + risk de regressions sin métrica real. Mejor stagger. |
| Shipear solo 1-2 esta sesión | El founder pidió "avanza todo". Lo entregable es 4 con 10 cases — honesto al estado experimental. |
| Crear un AgentRegistry para dispatch dinámico | Anti-pattern (R11): no Governor. Cada endpoint específico es más explícito. |
| Postergar M5.4 (sleeper) por complejidad financiera | El agente no inventa cifras (regla explícita en prompt). Razona desde cadence de filings, lo cual es legítimo. |
