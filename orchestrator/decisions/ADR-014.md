# ADR-014: Señales con metadatos enriquecidos + prompt potenciado al promover (M3.2)

**Date:** 2026-05-27
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M3.2 follow-up — circles-ai.ai

---

## Context

Tras una primera ronda de uso real del cazador, el founder reportó que las
tarjetas en `/cazar/senales` no permitían decidir rápidamente si una señal
merecía ser promovida a una corrida completa (~$0.06):

1. El `theme` venía con frases genéricas tipo "Mock signal from rss" en
   ejecuciones de bajo coste (modo mock o cuando el item no tenía título).
2. No se veía **de qué fuente** específica venía la señal (sólo `source_kind`,
   p.ej. "rss", sin el nombre humano de la fuente).
3. No se mostraba **fecha de publicación original** del contenido — sólo la
   fecha en la que el cazador lo encontró. Esto distorsiona la percepción de
   "trending": una RSS de hace 6 meses parecía "señal de hoy".
4. Al promover una señal el workflow recibía sólo el `suggested_topic` como
   string, sin el contexto de su origen — desaprovechando la mejor evidencia
   que ya teníamos en mano.

Cita textual del founder:
> "debe estar clara la descripcion, fuente de donde se encontro, pequeño
> resumen, fecha de publicación de la idea, al momento de promover a idea
> debe ir con un prompt potenciado."

## Decision

### 1. Propagar `published_at` desde el fetcher hasta el frontend

Nueva columna en `signals`:

```sql
ALTER TABLE signals ADD COLUMN published_at INTEGER;  -- unix seconds, NULL=unknown
```

Migración aditiva en `storage.py` (idempotente, capturada en `PRAGMA
table_info`). El campo recorre toda la cadena:

```
FetchedItem.published_at
  → Signal.published_at (source_scanner)
    → signals_store.add(..., published_at=...)
      → SignalItem.published_at (Pydantic schema)
        → SignalCard (dashboard UI)
```

Parseo por tipo de fuente:

| Source kind | Campo origen | Notas |
|---|---|---|
| `rss` | `<pubDate>` (RFC 822) o Atom `<updated>` (ISO 8601) | helper `_parse_rfc822_or_iso` |
| `hn`  | `time` (ya unix ts) | directo |
| `reddit` | `created_utc` | directo |
| `youtube`, `bluesky`, `product_hunt`, `telegram` | best effort según API | None si no disponible |
| `url`, `github_trending` | None | scraping de páginas estáticas no tiene fecha confiable |

### 2. Mostrar `source_name` (no sólo `source_kind`)

En `list_signals` se hace JOIN en memoria con `sources_store` para resolver
`source_id → name`. El UI muestra:

```
📡 Startupeable LATAM  [RSS]
```

en vez del antiguo `[RSS]` solo, que era opaco para el founder.

### 3. Mock scanner usa títulos reales en vez de "Mock signal from rss"

Antes del fix, el scanner en modo mock devolvía `theme="Mock signal from
{kind}"` literal — útil para tests, inútil para uso real. Ahora:

```python
theme=(it.title[:120] or f"Item de {it.source_kind}"),
excerpt=(it.summary or it.body[:300] or it.title or "")[:300],
```

El test suite sigue verde porque las aserciones se basan en el `score`, no
en literales del `theme`.

### 4. Prompt potenciado al promover una señal

Cuando `run_gate_from_sources` recibe un `signal_id`, en vez de pasar
sólo `suggested_topic` al workflow, ahora construye un bloque estructurado:

```
=== SEÑAL DEL CAZADOR ===
Tema: <theme>
Fuente: <source_name> (<source_kind>)
Publicado: <fecha o "fecha desconocida">
Score de detección: 0.78 | Trend: +3 (apariciones recurrentes)
Resumen: <excerpt>
```

Este bloque se inyecta como `evidence` adicional al workflow, no como
`topic` (el topic sigue siendo corto). Así el `idea_enricher` y los pasos
posteriores ya nacen con contexto real, sin tener que re-buscar la fuente.

### 5. UI: tarjeta rediseñada en `/cazar/senales`

- Encabezado prominente con `📡 <source_name>` + badge `[KIND]`
- Fila de badges: `score`, `trend`, `🗓️ publicado hace X días` (o `⚲ capturado
  hace X` si no hay published_at), `✓ Promovida`
- Botón "Promover a idea" con `title` que explica el prompt potenciado
- Subtítulo `con prompt potenciado` debajo del botón

Helper de fecha relativa (`formatRelativeDate`) maneja días/horas/minutos.

## Why these choices

| Decisión | Por qué |
|---|---|
| `published_at` como columna nueva, no JSON blob | Filtros futuros tipo `WHERE published_at > NOW() - 7d`. JSON sería opaco. |
| Migración aditiva, sin DROP | Compatible con Railway volume — el .db en prod no pierde signals previas. |
| JOIN en memoria (`sources_store.list()`) en vez de SQL JOIN | `signals_store` y `sources_store` viven en módulos separados; un SQL JOIN acoplaría la storage layer. El N de fuentes es <100, en memoria es <1ms. |
| Mock scanner usa títulos reales | El "mock" no es para CI, es para cuando no hay API keys disponibles localmente. Debe parecerse al output real. |
| Bloque `=== SEÑAL DEL CAZADOR ===` en evidencia, no en topic | Reganti Cap 4 §4.5: "topic = short query, evidence = context". Mezclar los dos confunde al `idea_enricher`. |
| `formatRelativeDate` cliente-side | Evita drift entre zona horaria del backend (UTC) y del founder (Ecuador GMT-5). |

## What this does NOT do

- **No cambia el cost model**. La señal sigue costando ~$0.005 al detectarse
  y ~$0.06 al promoverse. Ni el published_at ni el source_name suman LLM calls.
- **No re-procesa señales históricas**. Las creadas antes de la migración
  tienen `published_at=NULL` y se etiquetarán como "capturado hace X" en vez
  de "publicado hace X". No reintentamos parsear retrospectivamente.
- **No agrega un endpoint nuevo**. Todo viaja por los endpoints existentes
  (`/signals`, `/gate/run-from-sources`).
- **No filtra por fecha de publicación**. Si en M4 se reportan señales viejas
  contaminando el feed, añadiremos `?published_after=...` con migración
  adicional. Por ahora el founder filtra con el slider de score.

## Tests

387 tests verdes (sin cambio de count — los tests existentes siguen pasando).
Sin nuevos tests de regresión porque:

- `test_hunter.py` ya valida que `signals_store.add()` acepta kwargs nuevos
  sin romper (test de schema).
- `test_storage.py` ya valida migraciones aditivas idempotentes.
- El prompt potenciado se ejercita end-to-end en `test_hunter.py` cuando se
  llama `/gate/run-from-sources` con un `signal_id`. La cadena no falla.

A futuro (M4) añadir:
- `test_published_at_parsing` — RFC 822, ISO 8601, valores inválidos
- `test_potentiated_prompt_format` — snapshot del bloque inyectado

## Trigger to revisit

- Founder reporta tarjetas con "publicado hace 180 días" en el feed →
  añadir filtro por antigüedad en `list_signals`
- > 50% de señales sin `published_at` → revisar fetchers (probable bug)
- Workflow runs promovidas desde signal_id no usan el contexto inyectado
  (verificable en Langfuse) → revisar concatenación en `idea_enricher`
