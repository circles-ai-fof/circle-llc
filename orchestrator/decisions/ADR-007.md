# ADR-007: Specificity Refinement Loop (idea_hunter ↔ idea_enricher)

**Date:** 2026-05-26
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M1.6 — circles-ai.ai

---

## Context

M1.5 introduced `idea_enricher` to score specificity and sharpen vague IdeaSpecs. But the architecture was **one-shot**: enricher sharpened once and passed the result downstream regardless of whether the score crossed the 3.5 threshold.

Problem observed in live runs: very vague topics ("plataforma con IA") still produced score 2.8 → passed downstream as-is → maturer/validator/landing all worked with the under-specified idea → the gate result was noisy.

User asked for an iterative loop where hunter and enricher cooperate until specificity is good enough.

## Decision

Add a **bounded refinement loop** in `EvidenceGateWorkflow`:

```
for attempt in 1..3:
    idea = idea_hunter.generate(topic, feedback=previous_notes)
    refined, meta = idea_enricher.enrich(idea)
    if meta["specificity_score"] >= 3.5 or attempt == 3:
        break
    previous_notes = meta["refinement_notes"]
```

**Bound**: hard cap N=3 attempts. Never iterate forever (R13 budget tracker also enforces step cap).

**Termination criteria**:
1. `specificity_score >= 3.5` (success), OR
2. `attempt == 3` (give up; downstream consumes the best attempt; log warning), OR
3. Budget tracker raises `TrajectoryBudgetExceededError` (very rare; covered by R13)

## Why a loop and not multi-LLM debate

The loop is between **two different specialized agents** with distinct scopes:
- `idea_hunter` produces ideas
- `idea_enricher` evaluates and refines

This is **NOT** the multi-LLM debate anti-pattern (ADR-005). The loop has:
- A clear termination criterion (score ≥3.5)
- A hard step cap (N=3)
- Different agents at each step (specialization, not echo chamber)
- Deterministic test coverage (mock mode goes 1 iteration always)

## What this changes

- `orchestrator/agents/idea_hunter.py`: `generate(topic, feedback=None)` — new optional `feedback` param that, when set, tells the LLM what to fix.
- `orchestrator/workflows/evidence_gate.py`: replaces `idea = self._idea_hunter.generate(topic)` + `idea, meta = self._idea_enricher.enrich(idea)` with the loop above. Stores `enrichment_attempts` count in run metadata.
- `orchestrator/core/models.py`: optional `enrichment_attempts: int` on `EvidenceGateRun`.

## What this does NOT change

- The 6-step linear chassis from outside view (hunter+enricher is still "step 1.5", just now potentially 2-3 internal iterations).
- Downstream steps (maturer, validator, landing, gate) — they receive a single best IdeaSpec.
- Mock mode: deterministic, 1 attempt always.

## Consequences

**Positive**
- High-specificity ideas reach maturer → cleaner downstream, less noise in gate decision.
- The loop is observable: `enrichment_attempts` field tells founders if "this idea needed work".

**Negative**
- Cost per vague-idea run: +$0.01–0.02 per extra hunter call (acceptable, capped at +$0.04).
- Latency per vague-idea run: +5–10s when N=3 (only when input is vague).

## Trigger to revisit

- 95% of runs hit N=3 → enricher's bar is too high, or hunter is too vague.
- <5% of runs need >1 attempt → loop isn't earning its keep; remove or relax.
