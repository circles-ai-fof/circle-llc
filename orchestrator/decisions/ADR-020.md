# ADR-020: On-Demand Translation of Foreign-Language Signals (M4.4)

**Date:** 2026-05-28
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M4.4 — el founder pidió: *"dependiendo del idioma que esté en señales debe ser traducido al español o que exista la opción traducir"*.

---

## Context

El cazador captura señales de fuentes globales (Hacker News, GitHub trending,
RSS internacionales, Bluesky, Telegram). Una porción significativa de las
señales viene en inglés (y eventualmente otros idiomas), lo que obliga al
founder a saltar entre el dashboard y un traductor externo para entender de
qué trata cada idea.

Necesitamos:

1. **Detectar el idioma automáticamente** en el momento de la captura (sin LLM,
   para no inflar costos).
2. **Mostrar el idioma** en la UI para que el founder pueda decidir si quiere
   traducir.
3. **Traducir bajo demanda** (no auto-traducir todo, eso quemaría tokens en
   señales que el founder ni va a abrir).
4. **Persistir la traducción** para no re-traducir cada vez que se refresca.
5. **Permitir alternar** entre versión traducida y original (para verificación
   y para nombres propios).

## Decision

### 1. Detector heurístico (sin LLM) — `orchestrator/core/language.py`

Función `detect_language(text) -> (lang, confidence)`:

- Cuenta tokens de un set de marcadores en español vs inglés.
- Caracteres acentuados (`ñ`, `á`, `é`, `í`, `ó`, `ú`, `¿`, `¡`) suman +3
  cada uno — son señal fuerte de español.
- Si el texto es muy corto (≤2 tokens) → `unknown`.
- Confidence = `(hits_dominantes - hits_otros) / total` normalizado a `[0,1]`.

Costo: $0. Latencia: <1ms. Suficiente para el 95% de los casos
(R: "stay at the simplest level").

### 2. Almacenamiento — columnas en `signals`

Migración aditiva (`storage._SCHEMA` + `PRAGMA table_info` idempotente):

| Columna | Tipo | Notas |
|---|---|---|
| `language` | TEXT | "es" / "en" / "unknown". Se llena en `Storage.add()` automáticamente. |
| `translated_theme` | TEXT NULL | Llenado tras llamar al endpoint translate. |
| `translated_excerpt` | TEXT NULL | Idem. |

`Storage._signal_row_to_dict` calcula `language` al vuelo si está NULL
(para señales pre-M4.4).

### 3. Endpoint `POST /api/v1/signals/{id}/translate`

Tres ramas, en orden de precedencia:

1. **Ya está en español** → retorna `already_in_spanish=True`, `cost=0`,
   no llama a Claude. Esto evita el gasto fantasma cuando el detector
   marca `es` con alta confianza.
2. **Mock mode** (`_workflow._mock_mode=True`, sin `ANTHROPIC_API_KEY`)
   → retorna `"[Traducción demo] {theme}"` y `"[Traducción demo] {excerpt}"`.
   Permite que CI corra sin API key y que el founder pruebe el flujo UI
   sin gastar dinero.
3. **Real mode** → Claude **Haiku 4.5** (`claude-haiku-4-5`) con el prompt:
   > "Traduce este título y resumen al español neutro (LATAM). Mantén
   > nombres propios, marcas, productos y tecnologías sin traducir."
   Output: JSON `{theme, excerpt}`. Costo estimado ~$0.0005 por señal.

Tras éxito, persiste vía `Storage.set_translation(id, translated_theme,
translated_excerpt, original_language)`.

### 4. UI — `/cazar/senales` (listing) y `/cazar/senales/[id]` (detail)

- Botón **🌐 Traducir (EN)** aparece solo cuando `language !== "es"`
  y `!translated_theme`.
- Tras traducir, el botón se transforma en **🌐 Ver original (EN)** /
  **🇪🇸 Ver traducción** (toggle).
- Banner pequeño bajo el título: *"🌐 Traducido del EN al español"* /
  *"📘 Mostrando original (EN)"* para que el founder sepa qué está viendo.
- Por defecto se muestra la traducción cuando existe (lo más útil al hacer
  scroll rápido).
- Double-click guard en el handler (mismo patrón que `analyze`).

## Consequences

### Positivas

- El founder puede triagear señales en inglés sin cambiar de pestaña.
- Costo mínimo: $0 de detección + $0.0005 por señal traducida bajo demanda
  (no auto-traducir).
- Skipping cuando `lang == "es"` evita gasto fantasma en señales locales.
- Mock mode mantiene CI verde sin API keys.
- Schema aditivo — señales viejas (M3.x) muestran "🌐 Traducir (UNKNOWN)"
  solo si el detector las clasifica como no-español al vuelo.

### Negativas / aceptadas

- El detector heurístico clasificará mal idiomas no contemplados (PT, FR, DE)
  como `unknown` y no ofrecerá traducir. Decisión consciente: priorizamos
  ES↔EN (95% del tráfico actual). PT/FR llegará cuando aparezcan fuentes
  brasileras/francesas.
- Re-traducir requiere borrar la columna a mano vía SQL — no exponemos
  un endpoint "re-translate" hasta tener evidencia de necesidad.
- Nombres propios pueden traducirse mal pese al prompt — el toggle
  "Ver original" mitiga.

## Alternatives considered

| Alternativa | Por qué NO |
|---|---|
| Auto-traducir TODAS las señales en `scan()` | Quemaría ~$0.0005 × N señales/día sin garantía de que el founder vaya a abrirlas. ROI negativo. |
| Usar `langdetect` (lib externa) | +12MB de dependencia para algo que hago en 40 líneas con suficiente precisión. R: minimalismo. |
| Detección con LLM (Claude classify) | $0.001 × cada señal en cada scan. ROI catastrófico. |
| Traducir vía Google Translate API | Cuenta separada + token + facturación. Cero ventaja sobre Haiku que ya tenemos pagado. |

## Tests

  helpers).
- `tests/api/test_hunter.py::test_translate_*` — 4 tests:
  - skip cuando ya está en español
  - mock mode produce `"[Traducción demo]"`
  - 404 para signal_id desconocido
  - 401 sin Bearer token

Total: 103/103 verdes en el subset M4.4.

## Migration

Aditiva, idempotente. Schema se aplica en el próximo arranque del API.
Señales pre-M4.4 quedan con `language=NULL` y `_signal_row_to_dict` las
clasifica al vuelo en cada GET. No hay backfill batch — el costo es nulo
porque la detección es heurística.
