# ADR-015: IdeaAnalyzer — señales accionables (M3.5 + M3.6)

**Date:** 2026-05-27
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M3.5 + M3.6 — circles-ai.ai

---

## Context

Tras M3.4 el founder reportó que al revisar `/cazar/señales` no podía
decidir si una señal valía la pena promover. Cita textual:

> "las tarjetas no dicen mucho — no entiendo qué idea estoy a punto de
> promover. Necesito información para decidir, búsqueda, mercados
> potenciales que permitan tomar la decisión."

La promoción cuesta ~$0.06 cada una. Sin info accionable, el founder o
gastaba a ciegas, o no promovía nada. Necesitábamos un paso intermedio
de triage que costara menos y diera contexto.

## Decision

### 1. Nuevo agente `IdeaAnalyzer` (single-call, ~$0.005)

Toma una señal capturada y produce `SignalAnalysis`:

```python
@dataclass
class SignalAnalysis:
    market_size_estimate: str   # "~50k empresas LATAM" / "$2B TAM"
    icp_probable: str           # perfil de cliente (rol + tamaño + país)
    competitors: List[str]      # 0-5 nombres conocidos (o "ninguno")
    differentiator: str         # ángulo único (1 frase)
    risks: List[str]            # 2-3 riesgos principales
    recommendation: str         # promote | wait_for_more_data | discard
    reasoning: str              # 1-2 frases justificando
```

Mock-aware: sin `ANTHROPIC_API_KEY`, genera placeholders en español con
heurísticas mínimas (keywords "test"/"mock" → discard, "ai"/"ia" →
wait_for_more_data, resto → promote).

Persistido en `signals.analysis_json` (columna nueva, migración aditiva).

### 2. Endpoints

| Endpoint | Propósito |
|---|---|
| `GET /api/v1/signals/{id}` | Detail de una señal con analysis + item_titles |
| `POST /api/v1/signals/{id}/analyze` | Analiza UNA señal (~$0.005) |
| `POST /api/v1/signals/analyze-batch` | Analiza N en lote (explicit IDs o top_n por trend) |
| `POST /api/v1/signals/cleanup-mocks` | Borra "Mock signal from..." obsoletas |

`ScanRunRequest` gana `auto_analyze_trend_threshold` (0-10, 0=off). El
autoscan loop lee env `AUTO_ANALYZE_TREND_THRESHOLD` cada iteración:
cuando una señal nueva alcanza trend ≥ threshold, se enriquece al vuelo.

### 3. UI

**`/cazar/señales` (lista mejorada):**
- 2 filas de filtros: score · ordenar · fuente / búsqueda · feedback · trend mín
- Botones: 🔄 Re-escanear limpio · 🗑️ Borrar mocks · 🧹 Limpiar viejas · ↻ Refresh
- Botón "🤖 Analizar top 10" para batch
- Cada tarjeta: badge 🟢/🟡/🔴 con recomendación, URLs con título real
- Link "Ver detalle completo →"

**`/cazar/senales/[id]` (vista detalle):**
- Header con metadata
- Action bar: Up/Down · Re-analizar · Promover ($0.06)
- Panel de análisis estructurado (grid 2-col + riesgos)
- Topic sugerido destacado
- Items con título legible y URL clickable

### 4. Signal gana `item_titles` (List[str])

Paralelo a `evidence_urls` — guardamos el título de cada item del cluster
para que el dashboard muestre hovercards informativos en vez de hostnames.

## Why these choices

| Decisión | Razón |
|---|---|
| Agente separado en vez de extender source_scanner | Single-responsibility: scanner detecta, analyzer evalúa. El scanner ya corre en batch grande; analyzer se invoca on-demand por señal. |
| Persistir el análisis en signals.analysis_json | Si el founder vuelve a abrir la tarjeta, no se cobra LLM otra vez. Tampoco se mezcla con la storage de runs. |
| Recommendation enum `promote/wait/discard` | 3 opciones cubren 90% de decisiones reales. Más granularidad = parálisis. |
| Batch endpoint con top_n auto-pick | Founder con 50 señales no va a hacer 50 clicks. "Analizar top 10 por trend" es la operación natural. |
| `AUTO_ANALYZE_TREND_THRESHOLD` env-controlled | Algunos founders quieren todo manual, otros quieren automatización máxima. Env var = sin code change. |
| Botón "Re-escanear limpio" en UI vs endpoint nuevo | Orquestación en cliente = backend stateless. Reganti Cap 10 §10.3: "compose endpoints, don't create meta-endpoints". |
| Mock heurístico con keywords | Sin esto la UX en modo demo era todas "promote" — no ejercitaba los 3 caminos del UI. |

## What this does NOT do

- **No usa Sonnet** — Haiku-tier por costo. El análisis es "estructura + estimación", no requiere razonamiento profundo.
- **No genera el landing copy** — eso lo hace `landing_generator` después de promover.
- **No reemplaza al `gate_decider`** — el verdict pass/kill/iterate sigue siendo del workflow completo. `analyzer.recommendation` es un *triage previo* mucho más barato.
- **No filtra señales automáticamente** — ni siquiera con `recommendation=discard`. La decisión final es del founder. Sólo coloreamos.
- **No re-analiza al promover** — el análisis persiste en `analysis_json` y se usa como evidencia adicional al hunter (próximo M3.7).

## Cost model

| Operación | Costo |
|---|---|
| Una señal capturada (scanner) | $0 (rule-based mock o ~$0.001 LLM batch) |
| Analizar UNA señal | ~$0.005 (single Haiku call) |
| Analizar batch de 10 | ~$0.05 |
| Promover una señal a workflow completo | ~$0.06 |
| Auto-análisis durante autoscan (trend ≥3) | ~$0.005 cada |

Comparativa: si tienes 100 señales/semana y analizas las 20 con trend ≥2,
gastas $0.10/semana en triage. De esas, promueves las 5 más prometedoras:
$0.30/semana en gate runs. Total $0.40/semana = $20.80/año.

## Tests

- **6 hunter tests** para endpoints `/signals/{id}/analyze` + `/signals/cleanup-mocks` (passing in test_hunter.py)
- **30 golden cases** en `orchestrator/evals/golden_cases/idea_analyzer.jsonl`
- **34 smoke tests** en `tests/test_idea_analyzer.py`:
  - Shape + non-empty + recommendation-valid contra cada golden case
  - Determinism (same input → same output)
  - Discard heuristic for garbage themes
  - Langfuse trace emission with name `IdeaAnalyzerAgent._call`
- **8 batch + scan validation tests**: explicit IDs, skip already, auto-pick top_n, auth, threshold bounds

Total: **460 tests passing** (was 400 before M3.5).

## Observabilidad

Cada `_call()` de IdeaAnalyzer emite a Langfuse:
- `name="IdeaAnalyzerAgent._call"`
- input/output tokens
- model
- prompt + response

Filtrable por agente en el dashboard de Langfuse — el founder puede ver
cuántas señales se analizaron, costo total, latencia p95.

## Trigger to revisit

- **>30% de las recomendaciones del analyzer son contradichas por el gate_decider posterior** → el analyzer está sobre/sub-confiado, recalibrar prompt
- **Founder analiza la misma señal 3+ veces seguidas** → falta info crítica que el analyzer no captura, expandir SignalAnalysis
- **`AUTO_ANALYZE_TREND_THRESHOLD` lleva a >$50/mes de gasto** → poner cap diario
- **Botón "🤖 Analizar top 10" usado <1 vez/semana** → era complejidad innecesaria, considerar remover
