# ADR-018: Connected Accounts + Platform Detection (M4.0)

**Date:** 2026-05-28
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M4.0 — "El Cazador" Fase 1 completada

---

## Context

El founder pidió completar la Fase 1 del Cazador (M3 ya tenía 9 source kinds
y bulk-delete). Lo que faltaba:
> "Cuando pego un link de una fuente (Instagram, TikTok, LinkedIn, YouTube,
> blog/news), el sistema detecta la plataforma automáticamente. Si esa
> plataforma NO tiene cuenta conectada, me avisa: 'Falta conectar cuenta'."

Hoy aceptamos cualquier URL sin chequear credenciales — el founder añade
una URL de YouTube y descubre 10 min después en `/cazar/senales` que no se
cargó nada porque falta `YOUTUBE_API_KEY`.

## Decision

### 1. Tabla `connected_accounts` (anotación, NO credenciales)

```sql
CREATE TABLE connected_accounts (
  platform        TEXT PRIMARY KEY,
  status          TEXT NOT NULL,    -- 'ready'|'configured'|'requires_credentials'|'deferred'
  oauth_required  INTEGER NOT NULL DEFAULT 0,
  notes           TEXT,
  configured_at   INTEGER
);
```

**Importante**: esta tabla NO guarda passwords/tokens. Es solo un registro
de "el founder dijo que esta plataforma está configurada el día X".
Las credenciales reales viven en `orchestrator/.env` como env vars.

### 2. `orchestrator/core/platforms.py` — catálogo de plataformas

Hardcoded — porque las decisiones de qué soportar son de producto, no
de runtime. 14 plataformas catalogadas:

**Ready (sin creds)**:
- `rss`, `url`, `hn`, `reddit`, `bluesky`, `telegram`,
  `github_trending`, `product_hunt`

**Optional credentials** (funciona sin pero mejora):
- `youtube` (RSS por canal sin key, Data API con key)

**Deferred** (ADR-011):
- `x` ($100/mo API), `linkedin` (OAuth partner), `instagram`,
  `facebook`, `tiktok` (Graph APIs solo TUS posts)

### 3. Funciones públicas

- `detect_platform(url) -> str | None` — regex sobre hostname/path
- `check_credentials(platform) -> dict` — combina catálogo + env vars
- `list_all_platforms_status() -> list[dict]` — para /configuracion/cuentas

### 4. Endpoints nuevos

| Endpoint | Auth | Propósito |
|---|---|---|
| `POST /api/v1/sources/check-platform` | `_require_user` | UI lo llama con debounce 400ms cuando el founder pega URL. Devuelve detected platform + credential status + recommended_kind |
| `GET /api/v1/connected-accounts` | `_require_user` | /configuracion/cuentas page lista las 14 plataformas con su estado actual |
| `POST /api/v1/connected-accounts` | `_require_user` | Founder marca una plataforma como "configured" cuando agregó el API key al .env |

### 5. UI

**`/cazar/fuentes`** form de añadir fuente:
- Debounce 400ms al cambiar `target` field
- Llama `check-platform`, muestra banner amarillo/verde/rojo según status
- Sugiere cambiar el `kind` dropdown si `recommended_kind` ≠ kind actual
- Lista variables faltantes en `.env` con código copiable

**`/configuracion/cuentas`** página nueva:
- 4 secciones por status (ready/optional/requires/deferred)
- Card por plataforma con: estado, mensaje explicativo, variables
  configuradas/faltantes, fecha de configuración, botón "✓ Marcar
  como configurada"
- Las deferidas explican POR QUÉ están deferidas (link a ADR-011)

## Why these choices

| Decisión | Por qué |
|---|---|
| Catálogo hardcoded en `platforms.py` vs DB | Las decisiones de qué soportar son de producto, no de runtime. Cambiarlas requiere review (ADR), no UI |
| `connected_accounts` solo anotación, NO credenciales | Reduce blast radius si alguien lee el .db. Credenciales en env vars (Railway secrets en prod). Cifrar la BD entera vendrá en M5 si es necesario |
| Debounce 400ms en check-platform | Sensación de instantaneidad sin saturar al backend cuando el founder escribe rápido |
| Mostrar las deferidas explícitamente | El founder necesita VER las que NO vamos a soportar para entender por qué (no es un bug, es una decisión) |
| Status enum corto | 4 valores cubren 100% de los casos: ready, configured, optional_credentials, requires_credentials, deferred |
| `recommended_kind` separado de `platform` | "youtube" es la plataforma; "youtube" o "rss" es el kind a registrar. Un URL de YouTube de canal podría registrarse como RSS sin Data API |

## What this does NOT do

- **No almacena credenciales** — eso requiere ADR separado (M5 si llegamos a multi-tenant con users que añaden sus propias keys de YouTube)
- **No implementa OAuth flow** — los `oauth_required: true` están como TODO. Cuando se implementen, será un endpoint `/oauth/{platform}/callback` que valida el token y lo guarda en env-var-style
- **No activa YouTube Data API v3** — sigue siendo RSS por canal. La key se reconoce y reporta como "configured" pero `fetch_youtube_channel` no la usa todavía. M4.0.5 (opcional, no en este commit) lo activaría
- **No detecta cuál cuenta específica de Instagram/X tiene el founder** — la integración real requeriría OAuth (deferido)

## Cost model

Cero. `check-platform` es un grep de URL contra el catálogo + `os.getenv`.
No hay LLM call, no hay DB write (solo el upsert manual).

## Trigger to revisit

- **Founder pide auto-configurar OAuth** → implementar `/oauth/{platform}` flow
- **Founder añade plataforma no soportada** (Threads, Mastodon, ...) → ADR adendum extendiendo el catálogo
- **`check-platform` cae lento** → cachear el catálogo en memoria (hoy se construye cada call, ~100µs OK)
- **Multi-tenant** → tabla pasaría a `connected_accounts(user_email, platform, ...)`
