# ADR-026: M5.8–M5.11 — Promoción de los 4 agentes experimentales a ACTIVE

**Date:** 2026-05-30
**Status:** Accepted — los 4 agentes promovidos a active(on-demand) v1.0.0
**Deciders:** Cristian Molina
**Context:** ADR-023 estableció el lifecycle `experimental → active`. ADR-024
demostró el patrón con TrendGapAnalyzer (M5.0 → M5.1). Este ADR aplica el
mismo patrón a los 4 agentes restantes en una sola sesión.

---

## Context

Tras M5.7, los 4 agentes experimentales (NicheScout, EventRelevanceScorer,
SleeperCompanyDetector, ProductArbitrageEvaluator) tenían 10 golden cases
cada uno + endpoints funcionales + UI integrada. El paso natural es
completar a 30 cases y promover a active(on-demand).

## Decision

### 80 golden cases nuevos (20 por agente)

Cada agente recibe el mismo set de 20 categorías de tests:

| Categoría | Cuántos | Verifica |
|---|---|---|
| Determinism | 1 | dos llamadas iguales → mismo output |
| Schema validation | 4-5 | caps (max_length, ge/le), tipos, required fields |
| Range/calibration | 2-3 | confidence ∈ [0,1], boundaries, monotonicity |
| Recommendation logic | 2-3 | valores en set válido, lógica per-recommendation |
| Localization | 1 | español detectable (acentos o palabras clave) |
| Cost/mock invariants | 2 | cost=0.0 en mock, mock_mode=True |
| Response shape | 1 | dict keys == set esperado (no extras, no missing) |
| Robustness | 2-3 | inputs edge case (vacíos, max, chars especiales) |
| Domain-specific | 3-4 | reglas únicas del agente (ej: 'y combinator' → go) |

### Cambios en código (4 archivos similares)

Para cada agente:

```python
# ANTES
AGENT_VERSION = "0.1.0"
EXPERIMENTAL = True
"""Status: experimental (M5.x). 10/30 golden cases."""

# DESPUÉS
AGENT_VERSION = "1.0.0"
EXPERIMENTAL = False
"""Status: ACTIVE (M5.x+6). 30/30 golden cases. v1.0.0."""
```

## Consequences

### Positivas

- **Patrón replicado 5 veces** (M5.0→M5.1, M5.2→M5.8, M5.3→M5.9,
  M5.4→M5.10, M5.5→M5.11). El playbook es sólido — cualquier agente
  futuro sigue el mismo molde.
- **80 golden cases nuevos** sin regresiones. La suite total pasa de 613
  a 693 verdes.
- **Cero agentes en estado intermedio** — el founder puede usar los 5
  M5.x agents con confianza production-ready (badge "DEMO" sigue
  apareciendo cuando no hay API key, pero el código está versioned y
  estable).
- **Versionado semántico:** `v1.0.0` significa contrato estable. Si en
  M6+ cambia el shape del response será `v2.0.0` con migración explícita.

### Negativas / aceptadas

- Los 30 cases por agente son TODOS mock_mode. Cero cases live LLM. Esto
  es deliberado: cases live se generan de uso real del founder en
  producción (y se transforman en regression tests post-bug).
- Bump a v1.0.0 implica contrato — cambios futuros requieren migration.
  Para los 4 agentes, los responses siguen la estructura definida en
  M5.2-M5.5, sin cambios desde entonces.
- Algunos cases dependen de heurísticas del mock (ej: "Y Combinator" →
  GO en EventScorer). Si refactoreamos las heurísticas mock para que sean
  más realistas, esos cases pueden romper. Mitigación: están diseñados
  para verificar el COMPORTAMIENTO observable, no implementación interna.

## Alternatives considered

| Alternativa | Por qué NO |
|---|---|
| Promover los 4 en sprints separados (M5.8 → M5.9 → M5.10 → M5.11 con commits separados) | Más overhead administrativo sin valor adicional. Un commit consolidado documenta mejor el patrón. |
| Mantener todos como experimental hasta tener data live | El founder ya pidió usarlos. Mantener experimental crea expectativa de inestabilidad falsa. |
| Saltarse golden cases y promover directo | Viola R12 explícitamente. El lifecycle existe por una razón. |

## Migration

Aditiva — bump de version `0.1.0 → 1.0.0` en 4 agentes. El response
shape no cambia. Los endpoints siguen iguales. Frontend no requiere
cambios.

Único cambio observable externo: SCOPES.md ahora muestra los 5 agentes
M5.x como **ACTIVE** en una tabla unificada (vs separación
"experimental" antes).
