# ADR-016: Operacionalización del cazador (M3.7 + M3.8 + M3.9)

**Date:** 2026-05-27
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprints M3.7 + M3.8 + M3.9 — circles-ai.ai

---

## Context

Tras ADR-015 las señales son accionables pero el flujo aún tenía 7 fricciones:

1. Auto-promote solo por score, no por trend acumulado
2. El análisis previo de una señal se perdía al promoverla (el hunter lo re-hacía)
3. Sin export — la data vivía solo en la UI
4. Sin contadores en sidebar → founder revisaba dashboard para saber si había nuevo
5. Sin notificación cuando algo importante pasaba (founder no abre el dashboard si no recibe ping)
6. Mobile broken — sidebar fijo de 240px ocupaba todo el ancho del cel
7. Búsqueda solo cliente-side — no escala con >500 señales

## Decision

### M3.7 — Quick wins backend

| Cambio | Por qué |
|---|---|
| `auto_promote_trend_threshold` (env + ScanRunRequest) | Promover automáticamente cuando trend ≥ N. Costo: ~$0.06/run. 0=off (default) |
| `=== ANÁLISIS PREVIO (IdeaAnalyzer) ===` en evidence_parts | Cuando una señal con análisis se promueve, el bloque se inyecta como evidencia adicional al hunter. Evita re-hacer ICP/mercado/competencia |
| `GET /api/v1/signals.csv` | Streaming CSV completo (id, fecha, fuente, score, trend, feedback, run_id, recomendación, mercado, ICP, URLs). Filtro `promoted_only=true` |
| `GET /api/v1/stats` | Endpoint único para badges del sidebar — counts agregados de signals/sources/runs + costo acumulado |
| `lifespan` replaces `on_event` | FastAPI deprecó on_event. Migración a `contextlib.asynccontextmanager` con `app.router.lifespan_context` |
| `datetime.now(timezone.utc)` replaces `datetime.utcnow()` | 96 deprecation warnings → 3 (los restantes son de Pydantic/Starlette) |

### M3.8 — LLM-judge + webhooks + mobile

**LLM-judge para IdeaAnalyzer** — cierra R12 estricto:
- `rubric_idea_analyzer.md` con 5 criterios (recomendación / mercado / ICP / riesgos / coherencia), escala 1-5
- 15 calibration cases en `calibration_dataset.jsonl` (`cal-051..cal-065`) — 6 promote / 4 wait / 5 discard
- `run_calibration.py._run_idea_analyzer()` ejecuta el agente real y compara contra human_label
- Target ≥80% agreement, mismo umbral que los otros agentes

**Webhooks fire-and-forget** (`orchestrator/core/webhooks.py`):
- 5 event types: `signal.high_trend`, `signal.analyzed.promote`, `signal.auto_promoted`, `run.completed`, `run.pending_review`
- Env config: `WEBHOOK_URL_SLACK`, `WEBHOOK_URL_GENERIC`, `WEBHOOK_MIN_TREND`, `WEBHOOK_EVENTS`, `DASHBOARD_BASE_URL`
- Slack-compatible payload (con block-kit fallback a `text`)
- Cableado en `_serialize_run`, `analyze_signal`, `_run_scan_internal`
- Reganti Cap 12 §12.4: "fire-and-forget is the only acceptable pattern for outbound notifications" — usamos `urllib` con timeout=5s y `except: pass`

**Mobile responsive** (`AuthenticatedShell.tsx`):
- Hamburger button (md:hidden, z-40, fixed top-left) toggle drawer
- Backdrop semitransparente con cierre al click
- Sidebar pasa de `fixed` (mobile) a `static` (md+) sin breakpoint manual
- `Sidebar` acepta `onNavigate?` callback para cerrar el drawer al click en link
- Padding fluido en /cazar/* con `clamp(20px, 4vw, 32px) clamp(16px, 4vw, 40px)`

### M3.9 — Search server-side

| Cambio | Por qué |
|---|---|
| `SignalsStore.list(search=…)` con LIKE | Filtro server-side case-insensitive en theme + excerpt + suggested_topic |
| `?search=` query param en `GET /signals` | Length-bound a 200 chars para evitar SQL slowdown |
| UI: input con debounce 350ms que llama `refresh({search})` | Antes filtraba cliente-side — ahora server-side via debounced fetch |
| Filter cliente-side queda SOLO para feedback + minTrend | Server hace el trabajo pesado, cliente refina lo cheap |

**No FTS5 todavía** — LIKE basta para <5k señales. Upgrade trivial cuando dataset escale:
1. Crear `CREATE VIRTUAL TABLE signals_fts USING fts5(theme, excerpt, suggested_topic, content=signals, content_rowid=id)`
2. Triggers `AFTER INSERT/UPDATE/DELETE` para sincronizar
3. Cambiar el `LIKE` por `MATCH ?` con `signals.id IN (SELECT rowid FROM signals_fts WHERE signals_fts MATCH ?)`

## Why these choices

| Decisión | Por qué |
|---|---|
| LIKE en lugar de FTS5 ahora | Sin migración, sin triggers, sin entropía. Funciona <5k filas y se reemplaza en 1 commit. |
| Webhooks con stdlib `urllib` | Sin nueva dep (httpx ya está, pero requests-fire-and-forget es minimal). 30 líneas total. |
| Slack-compatible message es solo `{"text":"…"}` | block-kit es overkill para una notificación operacional. Si el founder quiere bonito, hace una integración a Zapier vía WEBHOOK_URL_GENERIC. |
| Mobile sin Tailwind breakpoints custom | Usa `md:` de Tailwind ya configurado + `clamp()` CSS nativo. Cero CSS custom. |
| Search debounce 350ms | <300ms se siente "fluido", >500ms "lento". 350 es el sweet spot para conexiones LATAM mixtas. |
| stats endpoint es síncrono | Counts son <50ms en 5k filas. Un endpoint vs polling parallels = menor superficie de bugs. |

## What this does NOT do

- **No persiste el costo histórico** — runs viven en memoria, se pierden al reiniciar. M4 lo arregla con Outcome DB.
- **No tiene rate limit de webhooks** — si fueras a ejecutar 1000 scans/min, podrías saturar Slack. Cap viene en M4 con Redis.
- **No tiene Slack block-kit rich formatting** — el `{"text": "…"}` es funcional pero feo. Si lo necesitas, mapea WEBHOOK_URL_GENERIC a Zapier y formatea ahí.
- **No tiene FTS5** — preparado para upgrade pero no construido.
- **No tiene mobile-responsive de fuentes/pipeline/leads/etc** — solo `/cazar` y `/cazar/señales*`. Otras pages funcionan en mobile pero sin layout optimizado.

## Cost model

| Operación | Costo nuevo (M3.7-M3.9) |
|---|---|
| auto-promote por trend (1 señal) | ~$0.06 (igual a manual) |
| Webhook fire | $0 (network only, no LLM) |
| CSV export (5k filas) | $0 (1 query + stream) |
| /stats call | $0 (4 queries + aggregation) |
| Search query con LIKE | $0 (sigue siendo 1 query) |

## Observability

- `/diagnostic` ahora reporta `webhooks_configured: bool`
- `/stats` muestra `cost_usd_total_all_time` para visibilidad en sidebar
- Cada webhook fire se logguea a nivel INFO (success) / WARNING (failure)
- IdeaAnalyzer sigue emitiendo trace a Langfuse con nombre `IdeaAnalyzerAgent._call`

## Trigger to revisit

- **>5000 señales en la BD** → migrar search a FTS5 (1-2 horas de trabajo, ya documentado arriba)
- **>10 webhooks/min** → poner rate limit + Redis queue (M4)
- **Calibration agreement <80%** en idea_analyzer → revisar rubric criterion 1 o ajustar prompt
- **Mobile bounce rate >40%** → optimizar pipeline/fuentes pages también
- **Founder no usa /signals/search** → era complejidad innecesaria, considerar quitar
