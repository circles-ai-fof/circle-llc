# ADR-008: Cross-LLM Fact-Check on Numerical Claims

**Date:** 2026-05-26
**Status:** Accepted
**Deciders:** Cristian Molina
**Context:** Sprint M1.6 — circles-ai.ai

---

## Context

`idea_enricher` produces problem_statements with numerical claims ("60% PYMEs Ecuador esperan 90+ días para cobrar facturas"). These numbers are persuasive but **unverified by any source**. A vague-but-honest output is preferable to a sharp-but-fabricated one because downstream agents (maturer, validator, landing) anchor their work on those numbers.

The user's instinct was correct: "AIs should ask each other when uncertain". But for **factual claims**, the right pattern is NOT debate between models — it's **cross-LLM corroboration** where a second model checks if a claim is plausible/sourced (Wang et al. 2024, "Fact-Checking with Distant Supervision").

## Decision

After `idea_enricher` produces a sharpened IdeaSpec, **extract numerical claims** from `problem_statement` and `proposed_solution`, and **fact-check each one with Gemini** (cheapest provider, distinct training corpus).

Verdicts per claim:
- `SUPPORTED`: claim is plausible given general market knowledge
- `UNSUPPORTED`: claim contradicts known data or is implausible
- `NEEDS_VERIFICATION`: claim is too specific/niche for the checker to evaluate

Aggregate impact on the IdeaSpec:
- Each `UNSUPPORTED` claim **lowers** `specificity_score` by 0.5
- Two or more `UNSUPPORTED` claims → re-trigger the refinement loop (ADR-007) with feedback "these specific claims couldn't be corroborated; replace with verifiable ranges"

## Why this is not multi-LLM debate

| Multi-LLM debate (anti-pattern) | Cross-LLM fact-check (this ADR) |
|---|---|
| Both models argue about a classification | One model fact-checks the other's specific claim |
| Free-form back-and-forth | Single round, binary output (SUPPORTED / not) |
| No termination criterion | Hard structure: extract → check → score |
| Compounds biases | Different training corpora reduce shared bias |
| 5-10× cost | <$0.005 per check (Gemini Flash is cheap) |

The key distinction: **judgment vs. fact**. Models disagreeing on judgment is noise. A second model spotting a hallucinated number is signal.

## What this changes

- `orchestrator/core/fact_checker.py` — new module:
  - `extract_claims(text) -> list[Claim]` (regex + LLM, depending on mode)
  - `check_claim(claim) -> FactCheckResult` (calls Gemini via existing `_vote_gemini` infra)
- `orchestrator/agents/idea_enricher.py`: after sharpening, runs fact-check, attaches results to metadata, adjusts `specificity_score` per ADR rules.
- `orchestrator/core/models.py`: `EnrichmentMetadata` includes `fact_check_results: list[FactCheckResult]`.

## Activation rules

- **Enabled when**: `FACT_CHECK_ENABLED=true` AND `GOOGLE_API_KEY` set AND not in `mock_mode`
- **Disabled otherwise**: enricher behaves as M1.5 (no fact-check, no score adjustment)
- **Mock mode**: deterministic — returns one canned `SUPPORTED` and one `UNSUPPORTED` claim for predictable tests

## Cost & latency

- 1 Gemini Flash call per ~2 claims found = ~$0.001 per claim
- Typical run: 2-4 claims → +$0.002–0.004 per workflow
- Latency: +2-4s per run

## Consequences

**Positive**
- Detects hallucinated numbers BEFORE they propagate to landing copy & ad targeting
- Forces enricher to use **ranges** when uncertain ("between 30-50%") instead of false precision ("47.3%")
- Builds a corpus of UNSUPPORTED claims over time → useful as a model-evaluation dataset

**Negative**
- Gemini has its own knowledge cutoff and biases. False UNSUPPORTED verdicts will happen → log + sample for review.
- Adds ~10-15% to typical run latency for marginal grounding gain. Justified only because gates depend on these numbers.

**Risks**
- **Confirmation bias from same source**: if both models trained on the same web corpus, fact-check is weak. Mitigation: only Gemini (not Claude) as checker, since Gemini's pretraining is more distinct.
- **Over-rejection of niche-but-true claims**: ranges of N from the LATAM SMB space are often unknown to large models → would flag as UNSUPPORTED a true number. Mitigation: `NEEDS_VERIFICATION` is a soft warning, not a kill.

## Trigger to revisit

- >40% of claims flagged UNSUPPORTED on a calibration set of known-true claims → checker is too strict, retire.
- <2% claims flagged UNSUPPORTED across 50+ runs → not catching hallucinations, retire.
